# Outerstellar Framework - Implementation Plan

## Goal Description
The purpose of the Outerstellar Framework is to provide a clean, headless, framework-agnostic set of Kotlin utilities for modern JVM development. 
These libraries are designed to be imported as standalone Maven dependencies into any host application (such as Http4k web servers, Javalin microservices, JavaFX desktop clients, or Swing legacy apps) to provide foundational enterprise features like translation parsing, CSS variable calculation, and safe third-party extension loading via the standardized Java SPI (ServiceLoader) mechanism.

By decoupling these services from any specific framework, Outerstellar ensures maximum reusability across the entire enterprise ecosystem.

## Java Interoperability
While the libraries are written entirely in 100% Kotlin, they are designed to be explicitly compatible with legacy or pure-Java codebases. The framework intentionally utilizes interoperability annotations to ensure that a Java developer consuming `outerstellar-i18n` or `outerstellar-theme` in a pure Java Spring Boot or Java Swing application experiences a flawless, native-feeling API without any awkward Kotlin syntax leakage:
- **`@JvmOverloads`**: Forces the Kotlin compiler to generate native, overloaded Java methods automatically for any functions utilizing Kotlin's default parameters.
- **`@JvmStatic`**: Ensures that Kotlin `object` singletons or `companion object` factories expose standard `public static` methods to Java callers.
- **`@JvmName`**: Forces strict control over the generated bytecode accessor names, preventing awkward implicit Kotlin naming patterns from polluting the Java API.
- **Automatic Nullability Annotations**: The Kotlin compiler automatically injects `@NotNull` and `@Nullable` into the compiled `.class` files natively for every String, Int, or Object declared. When Java developers inject our services into their IDE (like IntelliJ or Eclipse), their IDE will natively warn them if they pass a null value to a strict Kotlin `String` parameter.

## Architecture & Sub-Modules

### 1. `outerstellar-bom` (Bill of Materials)
**Purpose:** The Root Parent POM managing dependency versions for the entire suite.
- **Design:** Aggregates `outerstellar-i18n`, `outerstellar-theme`, and `outerstellar-plugin`.
- **Value:** Prevents version conflicts for consuming developers. They declare the BOM once, and can import any Outerstellar library without worrying about version mismatching.

### 2. `outerstellar-i18n` (Translation Engine)
**Purpose:** A pure Kotlin wrapper over `java.util.ResourceBundle`.
- **Language Agnostic Syntax:** Introduces Spring-style `{0}`, `{1}` parameter replacement for strict compatibility and familiarity for enterprise developers.
- **Hot-Reloading Engine:** Circumvents the permanent caching mechanism of the native Java `ResourceBundle` using custom InputStreams, allowing developers to see translation changes in their web apps instantly during development without restarting the JVM.
- **Dependency Constraint:** Zero third-party dependencies outside of the Kotlin Standard Library.

### 3. `outerstellar-theme` (Style Engine)
**Purpose:** An agnostic color parsing engine that transforms JSON style definitions into multiple delivery formats.
- **Smart Shading:** Automatically computes lighter (`hover`) and darker (`pressed`) hex variants dynamically based on base configuration colors using algorithmic RGB shifting.
- **Web UI Delivery:** Emits the calculated colors as valid CSS Variables (`:root { --color-primary: #hex; }`) for injection into HTML `<head>` tags.
- **Desktop UI Delivery:** Exposes the raw hex maps for simple "Adapter" integration into FlatLaf (Swing) or JavaFX, ensuring perfect visual synchronization across entirely different tech stacks.
- **Dependency Constraint:** Requires `jackson-core` for robust JSON parsing.

### 4. `outerstellar-plugin` (SPI Extension Engine)
**Purpose:** A lightweight Kotlin abstraction layer over the native Java `ServiceLoader`. It allows host applications to safely discover and instantiate third-party compiled `.jar` extensions at runtime.
- **Isolation by Scope:** Allows the defining of custom entry point interfaces (e.g., `ServerPlugin`, `DesktopPlugin`), ensuring Web plugins cannot mistakenly inject code into Desktop runtime pipelines.

### 5. `outerstellar-exposed-plugin` (Optional Schema Generator)
**Purpose:** An optional bridge library integrating `outerstellar-plugin` with **JetBrains Exposed**.
- **The Data Definition Contract:** Instead of plugins dictating storage structures by running raw SQL `CREATE TABLE` scripts (a massive security and stability risk), plugins define their required data shape by providing pure Exposed `Table` objects or Kotlin Data Classes.
- **Host Application Sovereignty:** The Host Application (which imports this library and loads the plugins) remains in absolute, sovereign control of how that data definition is physically stored. 
  - A Web Server host may use this library to dynamically generate perfectly typed Native PostgreSQL tables based on the plugin's shape.
  - A lightweight Desktop host may ignore the database engine and instead serialize the plugin's data shape entirely to a local JSON file or in-memory map.

## Verification Plan

### Automated Tests
- Build dedicated JUnit 5 testing contexts within each module.
- For `outerstellar-i18n`: Verify parameter hydration correctly places Strings and Integers inside `{0}` blocks.
- For `outerstellar-theme`: Verify that the dynamic RGB shader correctly generates lighter/darker CSS strings from a static JSON file.
- For `outerstellar-plugin`: Mock a fake plugin `.jar` and verify `ServiceLoader` can automatically initialize the interface.

### Build Verification
- Execute `mvn clean install` at the root of the `C:\Develop\Claude\outerstellar\` BOM to ensure all modules compile, package into `.jar` files, and successfully deploy to the developer's local `~/.m2` Maven repository.
